# Docs scaffold init prompt

This file records the prompt used to generate the initial `/docs` structure, for reproducibility.

> Scaffold: **docs-scaffold-saas** · generated by `backlog_bandit/scripts/scaffold_docs.py`

## Prompt

Set up foundational documentation for a SaaS project. Markdown-first, agent-friendly.

### Structure created

```
/docs
├── README.md                          # Entry point + index of all docs
├── init-prompt.md                     # This file (the prompt that created the scaffold)
├── product/
│   ├── README.md                      # Index of product docs
│   ├── vision.md                      # What we are building, who for, why
│   ├── mvp-scope.md                   # What is in/out for MVP (incl. non-scope)
│   ├── roadmap.md                     # Now / Next / Later
│   └── features/
│       ├── README.md                  # Index of feature PRDs
│       └── _template.md               # PRD template
├── technical/
│   ├── README.md                      # Index of technical docs
│   ├── architecture.md                # System overview + diagram placeholder
│   ├── data-model.md                  # Entities, relationships, invariants
│   ├── api.md                         # API conventions; link to OpenAPI later
│   ├── adr/
│   │   ├── README.md                  # What an ADR is, how to add one
│   │   ├── _template.md               # Nygard format
│   │   └── 0001-record-architecture-decisions.md  # The "we use ADRs" ADR
│   └── design-docs/
│       ├── README.md                  # When to write a tech design doc
│       └── _template.md               # Tech design doc template
├── security/
│   ├── README.md                      # Index of security docs
│   └── baseline-checklist.md          # OWASP ASVS L1 checklist
└── reference/
    ├── README.md                      # Index of reference docs
    ├── full-documentation-set.md      # Exhaustive future list
    └── resources.md                   # Links to industry templates
```

### Templates used

- **PRD template** (`product/features/_template.md`): Summary, Problem/context, Target user, Goals, Non-goals, Success metrics, User flows, Scope, Open questions, Dependencies, Changelog
- **ADR template** (`technical/adr/_template.md`): Nygard format (Title, Status, Context, Decision, Consequences)
- **Design doc template** (`technical/design-docs/_template.md`): Summary, Background/motivation, Goals/non-goals, Proposed design, Alternatives considered, Risks/open questions, Rollout plan, Observability
- **Security checklist** (`security/baseline-checklist.md`): OWASP ASVS L1 categories with checkbox items

### Priority order for filling in

1. **product/vision.md** -- everything else flows from knowing what you are building and for whom
2. **product/mvp-scope.md** -- defines the boundary of Phase 1 work
3. **technical/architecture.md** -- establishes the system shape that all other technical docs reference
4. **technical/data-model.md** -- core entities drive API design, feature PRDs, and security decisions
5. **product/roadmap.md** -- sequence the work once scope is clear
6. **technical/api.md** -- conventions before you build endpoints
7. **security/baseline-checklist.md** -- review and check off items as you build (already populated with items, just needs status updates)
8. **product/features/*.md** -- write a PRD per feature as you start building it
